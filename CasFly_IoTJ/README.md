# CasFly — Setup and Reproduction Guide

CasFly is a decentralized causal chain tracing protocol for federated IoT health
networks. This guide explains how to install all dependencies, generate the
Synthea dataset from scratch, and reproduce every figure and table in the paper.

**No data files are distributed with this repository.** All data is generated
locally using the open-source Synthea patient simulator, which is freely
downloadable and requires no credentials.

## Generate the Synthea dataset

Synthea is a free, open-source synthetic patient generator that produces
realistic but entirely fictional EHR data. No sign-up or credentials are needed.

### 1. Install Synthea

```bash
# Java 11+ is required
java -version

git clone https://github.com/synthetichealth/synthea
cd synthea
```

### 2. Generate patients

```bash
# Generate 1000 patients (a quick run; paper used ~2243 patients)
./run_synthea -p 1000

# To match the paper patient count more closely:
./run_synthea -p 2243
```

Synthea writes CSV exports to `synthea/output/csv/`. The run takes a few minutes.

### 3. Copy the CSVs into the repo

Create the `raw_data/` folder and copy the three files the pipeline needs:

```bash
mkdir -p /path/to/CasFly/data/synthea/raw_data

cp synthea/output/csv/conditions.csv    /path/to/CasFly/data/synthea/raw_data/
cp synthea/output/csv/observations.csv  /path/to/CasFly/data/synthea/raw_data/
cp synthea/output/csv/immunizations.csv /path/to/CasFly/data/synthea/raw_data/
```

The folder should now contain:

```
data/synthea/raw_data/
├── conditions.csv       (patient diagnoses: condition name, start date, patient ID)
├── observations.csv     (lab results and vitals: date, value, units, patient ID)
└── immunizations.csv    (vaccination history: date, vaccine name, patient ID)
```

> **Why these three files?**
> - `conditions.csv` holds the clinical diagnoses (stroke, heart attack, etc.)
>   that CasFly uses as the starting events for causal chain queries.
> - `observations.csv` holds lab results and vitals (blood pressure, glucose, etc.)
>   which are the intermediate nodes that causal chains pass through.
> - `immunizations.csv` adds vaccination history as additional event types.
>   Together, the three files cover the full timeline of clinically observable
>   events for each simulated patient.

## Fix the path alias (one-time)

The pipeline scripts reference `data/synthea/raw_data/` which is the folder you
just created. If you placed the files somewhere else, update the `CONDITIONS_CSV`,
`OBSERVATIONS_CSV`, and `IMMUNIZATIONS_CSV` constants at the top of
`extract_cause_effect_pairs.py` and `filter_patients.py`.

All other intermediate paths are auto-created by the scripts themselves under
`data/synthea/processed/`, `data/synthea/causal_pairs/`, and `data/synthea/metrics/`.

Understanding why each intermediate file is created makes it easier to debug,
extend, or reuse parts of the pipeline independently.

    |            signal that the probability estimator consumes.

    |            1–3 months) and estimates P(effect | cause, lag_bin).

## Step-by-step reproduction

All commands are run from inside `casflyIoTJ/` unless noted.

```bash
cd /path/to/CasFly/casflyIoTJ
```

---

### Step 1 — Build probability tables

#### 1a. Extract temporal event pairs

```bash
python3 extract_cause_effect_pairs.py
```

Reads the three Synthea CSVs and emits every ordered (A before B) event pair
per patient. With 1000 patients expect ~500 000–2 000 000 rows depending on
how event-rich the generated population is.

Output: `data/synthea/processed/cause_effect_pairs.csv`

#### 1b. Compute conditional probabilities

```bash
python3 compute_conditional_probs.py
```

Groups pairs by (cause, effect, lag_bin) and computes
P(effect | cause, lag_bin) = count(cause -> effect in this lag bin) / count(cause).
The nine lag bins are: 0–7, 7–14, 14–30, 30–60, 60–90, 90–180, 180–365,
365–730, and 730–inf days.

Output: `data/synthea/processed/conditional_probabilities_all_events.csv`

#### 1c. Filter patients by study disease

```bash
python3 filter_patients.py
```

Keeps only patients with Stroke, Heart Attack (Myocardial infarction),
Lung Cancer, or Seizure disorder — the four conditions evaluated in the paper.

Output: `data/synthea/processed/filtered_patients.csv`

#### 1d. Build device lookup tables

```bash
python3 build_device_lookup.py
```

Assigns every (cause, effect) type pair to one of the 26 simulated IoT devices
using clinical keyword matching (e.g. blood-pressure-related events → BPCuff,
glucose-related → GlucoBand). Writes the PLTs that the wearable device scripts
use for inter-device query routing.

Outputs (all under `data/synthea/processed/`):
- `All_Device_Lookup_with_Probabilities.csv`
- `All_Device_Lookup_merged_with_ports.csv`
- and five companion cause/effect lookup CSVs

#### 1e. Partition patients to devices

```bash
python3 partition_patients_to_devices.py
```

Performs the same keyword-based assignment but writes per-device causal-pair
CSVs so each Device folder has its own local TPHG slice.

Output: `data/synthea/causal_pairs/Device_*_causal_pairs.csv`
        `data/synthea/processed/device_mapping.csv`

---

### Step 2 — Run the CasFly federation

This runs the actual distributed causal-chain protocol and produces the metrics
log that all downstream analysis consumes.

#### 2a. Create `filtered_patients_updated.csv`

This extends `filtered_patients.csv` with a `RELEVANT_DEVICES` column:

```bash
python3 - <<'PY'
import pandas as pd
patients = pd.read_csv("../data/synthea/processed/filtered_patients.csv")
patients["RELEVANT_DEVICES"] = ",".join([f"Device{i}" for i in range(1, 27)])
patients.to_csv("../data/synthea/processed/filtered_patients_updated.csv", index=False)
print(f"Wrote {len(patients)} rows")
PY
```

#### 2b. Understanding and running the individual device scripts

Each of the 26 devices is a standalone Python script located at:

```
data/synthea/device_scripts/
├── RenalWatch/
│   ├── RenalWatch.py
│   ├── custom_protocol.py
│   └── Patient_<uuid>.csv  (one CSV per patient assigned to this device)
├── AlkaSense/
│   └── ...
...
└── CasFlyHub/
    ├── CasFlyHub.py        (the chain initiator)
    └── custom_protocol.py
```

**What each Device*.py does internally:**

1. **Binds a UDP port** — each device reads its assigned IP and port from
   `All_Device_Lookup_with_Probabilities.csv` and opens a UDP socket via
   `custom_protocol.py` (a thin JSON-over-UDP wrapper).

2. **Loads the TPHG slice** — loads the per-device causal-pair CSV
   (`Device_N_causal_pairs.csv`) into a `networkx` directed graph where
   each edge `(cause_bin → effect_bin)` carries the conditional probability
   and lag bin as attributes. This is the Temporal Probabilistic Health
   Graph (TPHG) stored locally on the device.

3. **Starts a listener thread** — `handle_request()` runs as a daemon thread,
   blocking on `receive_packet()` until another device sends a JSON UDP
   packet containing the current chain state.

4. **Expands the chain on arrival** — when a packet arrives, the device calls
   `expand_causal_chain()`, which runs the LaVE algorithm (a Viterbi-based
   backward search with a priority queue) on its local TPHG to find the most
   probable causal predecessor. If the target event is not in the TPHG, LaVE
   falls back to a BFS search.

5. **Forwards or returns** — `forward_chain()` looks up the next device in the
   PLT by (cause_type, effect_type) and sends the updated chain state via UDP.
   When no further predecessor is found, `return_to_initiator()` sends the
   complete chain back to CasFlyHub on port 6019.

6. **Logs metrics** — `monitor_resources()` writes a per-device CSV
   (`<DeviceName>_<disease>_<patient>_metrics_log.csv`) capturing timestamp, chain
   path, timing breakdown (T, t1, t2, t_fallback), memory, CPU, and RAM.
   CasFlyHub then merges all per-device logs into `allmerged_metrics_log.csv`.

**Starting all listener devices (all except CasFlyHub):**

**Triggering CasFlyHub as the chain initiator (manual, single query):**

```bash
cd data/synthea/device_scripts/CasFlyHub
python3 CasFlyHub.py --initiate "Stroke" --patient_id "<patient-uuid>"
```

Replace `"Stroke"` with any of the four study diseases:
`Stroke`, `Myocardial infarction`, `Lung cancer`, `Seizure disorder`.
Replace `<patient-uuid>` with a UUID from `filtered_patients_updated.csv`.

What happens after you run this command:
1. CasFlyHub loads its TPHG and looks up the patient's observation history
2. It calls `expand_causal_chain("Stroke", patient_id)` — LaVE finds the most
   probable preceding cause in CasFlyHub's TPHG
3. `forward_chain()` consults the PLT to identify which wearable device owns
   that cause event and sends a JSON UDP packet to that device's port
4. The receiving device's `handle_request()` thread wakes up, expands the chain
   further, and forwards again
5. This continues until the causal chain is exhausted, then
   `return_to_initiator()` sends the final path back to CasFlyHub on port 6019
6. CasFlyHub prints the complete chain and writes the result to the metrics log

#### 2c. Launch the full experiment

```bash
bash run_experiment.sh
```

For each (patient, disease) row in `filtered_patients_updated.csv`:
1. Clears UDP port 6019 if already in use
2. Ensures all 23 listener wearable devices are running
3. Spawns CasFlyHub as the chain initiator
4. CasFlyHub queries its local TPHG, finds the most likely preceding cause,
   looks up which wearable device owns that cause in the PLT, and forwards the query
5. The chain propagates until no more causal predecessors are found (LaVE)
6. Results are appended to the metrics log

> **Runtime:** ~20 seconds per row (one patient/disease pair).
> With 25 patients across 4 diseases = ~33 minutes total.

- Runs PCMCI+ to discover statistically significant (cause -> effect) edges
- Extracts the edges that LaVE traced from `allmerged_metrics_log.csv`
- Computes precision, recall, F1, and Hits@K comparing the two edge sets

For each device and each tau, trains one logistic regression per effect bin

## Troubleshooting

**`conditions.csv not found` or similar**
Check that `data/synthea/raw_data/` exists and contains the three CSVs.
The scripts reference this exact path; update the constants at the top of
`extract_cause_effect_pairs.py` and `filter_patients.py` if your folder is named differently.

**`tigramite` import error**
Needs a C compiler:
- macOS: `xcode-select --install`
- Linux: `sudo apt install build-essential`
Then: `pip install tigramite`

**`allmerged_metrics_log.csv` not found**
This file is only created by running Step 2 (`run_experiment.sh`).
If you skip Step 2, you cannot run Steps 3 or 4 from scratch.
You must run `run_experiment.sh` at least once.

**Port already in use (UDP 6019)**
`run_experiment.sh` auto-clears the port. If it gets stuck:
```bash
lsof -nP -iUDP | grep 6019
kill -9 <PID>
```

**PCMCI+ is very slow**
Reduce scope for a quick sanity check: use `--top_n_devices 3 --taus 30`.
The `--resume` flag (default) caches per-tau PCMCI results so you only pay
the cost once per tau value.

**No significant PCMCI+ edges found**
With a small Synthea population (< 500 patients) some devices may not have
enough observations to satisfy PCMCI+'s significance threshold. Try generating
more patients (`./run_synthea -p 2000`) or lowering `--alpha 0.10`.

**Results differ from the paper**
This is expected when using a fresh Synthea run. The pipeline is correct;
the numbers will be consistent in direction (LaVE > PCMCI+ on AUPRC) but
the exact decimal values depend on the patient population.
